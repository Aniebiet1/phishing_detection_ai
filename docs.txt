Phishing Detection AI - Multi-Model Training and Evaluation Plan
Date: 2026-03-15

Goal
Build a robust, repeatable training workflow that compares multiple ML models on the same data split, then selects and saves the best model. Implement this in both:
1) train.py (script-first, automation-friendly)
2) Jupyter Notebook (experiment-first, visual analysis friendly)

You requested that training_report.json include multiple models and their evaluation. This plan does exactly that.

==================================================
1) WHAT WILL CHANGE
==================================================

A) train.py will be upgraded from single-model training to multi-model benchmarking.
B) A new notebook will be added (or an existing one can be created) to reproduce the same model comparison interactively.
C) training_report.json will become a structured report containing:
- dataset summary

D) The best model (based on a chosen metric) will be saved to saved_models/phishing_model.joblib for FastAPI inference compatibility.

==================================================
3) MODELS TO COMPARE (INITIAL SET)
==================================================

All models will use the same text vectorization pipeline (TF-IDF) for fairness.

Baseline comparison set:
1) LogisticRegression
2) LinearSVC
3) MultinomialNB
4) RandomForestClassifier
5) SGDClassifier (linear)

Why these models:
- LogisticRegression and LinearSVC are strong text baselines.
- MultinomialNB is often competitive and very fast.
- SGDClassifier scales well and can be tuned.
- RandomForest gives a non-linear alternative (usually weaker on sparse text, but useful as a comparator).

Optional future additions:
- XGBoost/LightGBM (if added to dependencies)
- Calibrated SVM for better probability estimates

==================================================
4) EVALUATION STRATEGY
==================================================

Single holdout split (initial implementation):
- train_test_split with stratify=label
- fixed random_state for reproducibility

Primary metric (for model selection):
- F1 score for phishing class (label=1)

Secondary metrics (always logged):
- Accuracy
- Precision (class 1)
- Recall (class 1)
- F1 (class 1)
- Macro F1
- Weighted F1
- Confusion matrix
- Inference support count
- Optional train/eval time per model

Why F1 for phishing class as primary:
- Phishing detection cares about balancing false negatives and false positives.
- Accuracy alone can hide class imbalance effects.

==================================================
5) train.py DETAILED IMPLEMENTATION PLAN
==================================================

Step 1: Refactor model construction
- Add a function like get_model_candidates() that returns a dictionary:
  {
    "logistic_regression": Pipeline(...),
    "linear_svc": Pipeline(...),
    "multinomial_nb": Pipeline(...),
    "random_forest": Pipeline(...),
    "sgd_classifier": Pipeline(...)
  }

Step 2: Standardized training/evaluation loop
- For each candidate model:
  1) fit on x_train, y_train
  2) predict on x_test
  3) compute metrics
  4) store metrics and metadata in results list

Step 3: Ranking and best-model selection
- Sort results by selection metric (default: phishing_f1 descending)
- tie-breakers (if needed): macro_f1 then accuracy
- select best model name + object

Step 4: Persist artifacts
- Save best model to --model-out path (default saved_models/phishing_model.joblib)
- Save full report to --report-out (default saved_models/training_report.json)

Step 5: Console summary
- Print compact ranking table (top to bottom)
- Print selected best model + key metrics

Step 6: CLI options enhancement
Add optional args such as:
- --selection-metric phishing_f1|accuracy|macro_f1
- --models logistic_regression,linear_svc,... (subset selection)
- --include-source-prefix (optional text preprocessing enhancement)

10) RISKS AND MITIGATIONS
==================================================

Risk: Some models do not expose predict_proba.
Mitigation: Use class predictions for metrics; probability-dependent metrics are optional.

Risk: Large dataset can increase training time.
Mitigation: Keep efficient defaults and allow --models to run subset quickly.

Risk: Label format inconsistencies.
Mitigation: Keep strict label normalization with clear error messages.

Risk: Report size grows.
Mitigation: Store concise summary metrics by default; include full classification report optionally.
